# 📦 Media Pipeline — Changelog

All notable changes to this project will be documented in this file.

This project follows **Semantic Versioning**.

---

# 🚀 [1.2.0] — Error System & Structured Failures

## ✨ Added

### 🔐 Structured Error System

Introduced a complete error hierarchy for predictable and type-safe error handling.

#### Base Error

```ts
class PipelineError extends Error {
  code: PipelineErrorCode;
  details?: Record<string, any>;
}
```

---

### 🧩 Error Types

Added specialized error classes:

* `ValidationError`
* `ProcessorError`
* `StorageError`

Each error:

* Extends `PipelineError`
* Includes a `code` field
* Supports structured `details`

---

### 🏷 Error Codes

```ts
type PipelineErrorCode =
  | "VALIDATION_ERROR"
  | "PROCESSOR_ERROR"
  | "STORAGE_ERROR"
  | "UNKNOWN_ERROR";
```

---

### 📤 Public Error Exports

Errors are now part of the public API:

```ts
import {
  PipelineError,
  ValidationError,
  ProcessorError,
  StorageError
} from "media-pipline";
```

---

### 🧠 Error Details Support

Errors can now carry structured metadata:

```ts
throw new ValidationError("File too large", {
  size: 5000,
  limit: 1000
});
```

---

## 🔧 Changed

### ⚙️ Executor Error Handling

The pipeline execution engine now:

* Wraps unknown errors
* Converts them into structured errors
* Preserves original error inside `details.originalError`

---

### 🔁 Validator Behavior

Validators now throw `ValidationError` instead of generic `Error`.

---

### 🔁 Processor Behavior

Processors are now expected to throw `ProcessorError` for controlled failures.

---

### 🔁 Storage Behavior

Storage layer errors are now wrapped as `StorageError`.

---

## 🛡 Backward Compatibility

✅ No breaking changes

* Existing pipelines continue to work
* Generic errors are automatically wrapped into structured errors
* Public API (`pipeline.process(file)`) remains unchanged

---

## 🧠 Architectural Impact

This upgrade enables:

* Clean API error responses (Next.js / Express)
* Reliable error type checking (`instanceof`)
* Centralized error handling strategies
* Debug-friendly pipelines
* Future logging & monitoring integrations

---

## 🔥 Example Usage

```ts
import { ValidationError } from "media-pipline";

try {
  await pipeline.process(file);
} catch (err) {
  if (err instanceof ValidationError) {
    console.log(err.code);     // VALIDATION_ERROR
    console.log(err.details);  // structured data
  }
}
```

---

## ⚠️ Notes for Contributors

* Use specific error classes instead of `Error`
* Always include useful `details` when possible
* Do not throw raw errors from validators/processors/storage
